A CSRF (Cross-Site Request Forgery) attack tricks an authenticated user into unknowingly executing unwanted actions on a web application, such as changing their password, updating their email, or transferring funds. How It Works Web applications frequently rely on cookies to remember that you are logged in. If you visit a malicious website while logged into another trusted site (like your bank or email provider), the malicious site can secretly instruct your browser to send a background request to the trusted site. Because your browser automatically attaches your session credentials (cookies) to any request sent to that trusted site, the server believes you explicitly intended to make the request. A Real-World Example You log into your bank account and leave the tab open. You click a suspicious link sent in an email, which takes you to the attacker's website.T he attacker's site contains a hidden form or a fake image that secretly triggers an HTTP request to your bank, instructing it to transfer money to the attacker's account.Because you are already authenticated, the bank accepts the transfer as legitimate. Common Defenses Developers protect applications from CSRF using a few key methods: Anti-CSRF Tokens: The server generates a unique, secret, and unpredictable token for the user's session. Every time the user submits a form, they must include this token. Because the attacker's site cannot predict the token, the request fails. SameSite Cookies: This cookie attribute prevents the browser from sending cookies on cross-site requests, blocking the attacker's ability to piggyback on your session. Re-Authentication: Websites often require users to type their current password again before allowing sensitive actions, such as changing login credentials or viewing billing details. ------------------------------------------------- If you've written HTML web forms in the past, you may have found it odd that there are no HTML fields in this template.
